# Pipeline Design

This document describes the design decisions, data flows, and interface contracts for each pipeline in the Parkinson Speech AI framework.

---

## Design Principles

1. **Configuration over code** — all pipeline parameters live in YAML config files, not in Python source.
2. **Reproducibility first** — random seeds, preprocessing hashes, and experiment configs are logged for every run.
3. **Shared interfaces** — both the traditional and transformer pipelines share the same dataset interface and evaluation module, enabling fair comparison.
4. **Lazy loading** — audio files are loaded on-demand during training to avoid memory exhaustion on large datasets.
5. **Pipeline composability** — each step in the pipeline is a self-contained unit that can be tested and benchmarked independently.

---

## Traditional ML Pipeline

### Step 1 — Data Loading
- `src/datasets/base_dataset.py` defines the abstract interface.
- Concrete dataset classes load a manifest CSV/JSON containing: `subject_id`, `label`, `file_path`, `sample_rate`, `metadata`.
- A single `get_item(index)` call returns a dict with `waveform` (numpy array), `sample_rate`, and `label`.

### Step 2 — Preprocessing
- Preprocessing chain is built from `src/preprocessing/pipeline.py`.
- The chain is fully parameterised by a config section `preprocessing:`.
- Processed audio is cached to `data/processed/` keyed by a hash of the preprocessing config to avoid redundant computation.

### Step 3 — Feature Extraction
- `src/features/pipeline.py` applies all enabled extractors in sequence.
- Each extractor returns a 1-D numpy array; the pipeline concatenates them into a single feature vector.
- Feature names and dimensions are logged for interpretability.

### Step 4 — Dataset Split
- Stratified k-fold splits are generated by `src/evaluation/cross_validation.py`.
- Splits are speaker-independent (no speaker appears in both train and test folds).
- Split indices are serialised to disk to ensure identical splits across models.

### Step 5 — Training
- `src/models/traditional/trainer.py` wraps scikit-learn's `Pipeline` (scaler + classifier).
- Hyperparameter tuning uses `GridSearchCV` or `Optuna` (configurable).
- The best model per fold is saved to `outputs/models/{experiment_name}/fold_{k}/`.

### Step 6 — Evaluation
- `src/evaluation/metrics.py` computes all metrics from fold-level predictions.
- Results are aggregated with mean ± std and bootstrap CIs.
- All metrics are logged to MLflow.

## Transformer Pipeline

### Step 1 — Data Loading
- Same dataset classes as the traditional path, but returning raw waveforms instead of feature vectors.
- A custom `collate_fn` in `src/datasets/collator.py` pads/truncates waveforms to a fixed length.

### Step 2 — Preprocessing
- Same preprocessing chain, but the output is a 16 kHz mono waveform (not features).
- The Hugging Face `AutoProcessor` (Wav2Vec2Processor, etc.) handles final normalisation and feature extraction to input values.

### Step 3 — Model Initialisation
- Pre-trained model loaded from Hugging Face Hub.
- A classification head (linear or 2-layer MLP) is appended on top.
- Two operating modes:
  - `frozen_encoder`: only the classification head is trainable.
  - `finetune`: all parameters are trainable with layer-wise learning rate decay.

### Step 4 — Training
- PyTorch training loop managed by Hugging Face `Trainer` or a custom loop in `src/models/transformers/trainer.py`.
- Mixed precision (fp16/bf16) training supported via `accelerate`.
- Gradient clipping, warmup scheduler, and early stopping are configurable.
- Checkpoints saved every N steps to `outputs/models/`.

### Step 5 — Evaluation
- Same `src/evaluation/metrics.py` module used for consistent metric computation.

---

## Cross-Corpus Evaluation Protocol

```
Train on: Italian Dataset → Test on: MDVR-KCL
Train on: MDVR-KCL       → Test on: Italian Dataset
```

- No fine-tuning on the target dataset.
- Evaluates domain generalisation without any adaptation.
- Results reported separately from within-corpus experiments.

## Config File Conventions

Each experiment maps to one YAML file in `configs/`. Example structure:

```yaml
experiment:
  name: wav2vec2_italian_finetune
  seed: 42

data:
  dataset: italian
  manifest_path: data/processed/italian_manifest.csv
  label_column: label
  subject_id_column: subject_id

preprocessing:
  target_sample_rate: 16000
  mono: true
  normalize: true
  trim_silence: true
  silence_threshold_db: -40
  max_duration_sec: 10.0

model:
  type: transformer
  backbone: facebook/wav2vec2-base
  mode: finetune  # frozen_encoder | finetune
  classification_head: linear
  num_classes: 2

training:
  epochs: 20
  batch_size: 16
  learning_rate: 1.0e-4
  weight_decay: 0.01
  warmup_ratio: 0.1
  gradient_clip_val: 1.0
  fp16: true
  early_stopping_patience: 5

evaluation:
  cv_folds: 10
  stratified: true
  speaker_independent: true
  bootstrap_n: 1000

tracking:
  mlflow_experiment: parkinson_speech_ai
  log_artifacts: true
```
